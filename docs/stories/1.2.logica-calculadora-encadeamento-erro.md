## Story

**As a** torcedor tricolor,
**I want** clicar nos botões da Calculadora Grêmio e obter resultados aritméticos corretos, incluindo encadeamento de múltiplas operações e tratamento de divisão por zero,
**so that** eu consiga realizar cálculos do dia a dia com confiança na correção dos resultados.

## Acceptance Criteria

1. **AC1.2.1:** Cada clique em um botão de dígito (`0`–`9`, identificados por `data-key="0".."9"`) anexa o dígito ao operando atual mostrado no display. **Comportamento de substituição:** se o display estiver mostrando `"0"` (estado inicial) OU um resultado de operação anterior OU `"Erro"` foi tratado por Clear, o novo dígito **substitui** o conteúdo do display em vez de anexar (evita `"05"` quando o usuário pressiona `5` no estado inicial). [Source: docs/prd.md#Requirements — FR1, FR2]

2. **AC1.2.2:** O botão de vírgula decimal (`data-key="."`) anexa um separador decimal ao operando atual. **Restrição:** múltiplas vírgulas no mesmo operando devem ser **ignoradas** — pressionar vírgula quando o operando já contém uma vírgula não tem efeito (`1,2,3` é proibido; deve resultar em `1,23`). Se o display estiver vazio ou em estado de "início de operando", a vírgula deve gerar `"0,"` (não `","` solto). [Source: docs/prd.md#Requirements — FR6]

3. **AC1.2.3:** Cada um dos 4 botões de operador (`data-op="add" | "subtract" | "multiply" | "divide"`) **registra** a operação pendente e prepara a calculadora para receber o segundo operando. O display **continua mostrando** o operando anterior (ou o resultado intermediário em caso de encadeamento) — **não** mostra o símbolo do operador isoladamente. [Source: docs/prd.md#Requirements — FR3]

4. **AC1.2.4:** O botão Igual (`data-action="equals"`) executa a operação pendente e exibe o resultado no display. **Cenários canônicos obrigatórios** (todos devem passar em validação manual):

   | Entrada (cliques sequenciais)         | Display final esperado |
   |---------------------------------------|------------------------|
   | `1` `+` `1` `=`                       | `2`                    |
   | `1` `0` `−` `3` `=`                   | `7`                    |
   | `4` `×` `5` `=`                       | `20`                   |
   | `1` `0` `÷` `4` `=`                   | `2,5`                  |
   | `0` `,` `1` `+` `0` `,` `2` `=`       | `0,3` (arredondamento na exibição é permitido para evitar ruído de ponto flutuante — ex: `parseFloat(result.toFixed(10))`) |

   Se Igual for pressionado **sem operação pendente**, o display permanece inalterado (operação no-op). [Source: docs/prd.md#Requirements — FR4]

5. **AC1.2.5:** **Encadeamento ATIVO de operações** — pressionar um operador após uma operação pendente já existente **resolve a operação anterior** e usa o resultado como primeiro operando da próxima. Avaliação **esquerda → direita**, **sem precedência matemática**. **Cenários canônicos obrigatórios**:

   | Entrada                          | Display final esperado | Justificativa |
   |----------------------------------|------------------------|---------------|
   | `2` `+` `3` `+` `5` `=`          | `10`                   | `2+3=5` → `5+5=10` |
   | `1` `0` `−` `2` `−` `3` `=`      | `5`                    | `10−2=8` → `8−3=5` |
   | `2` `+` `3` `×` `4` `=`          | `20`                   | `2+3=5` → `5×4=20` (sem precedência matemática — × NÃO tem prioridade sobre +) |
   | `1` `0` `÷` `2` `×` `3` `=`      | `15`                   | `10÷2=5` → `5×3=15` |

   Esta é a regra **central** da Story 1.2 e foi decisão **explícita** do sponsor (Brief Decisão Confirmada #1). [Source: docs/prd.md#Requirements — FR4 + docs/project-brief.md#decisoes-confirmadas-1]

6. **AC1.2.6:** **Divisão por zero** exibe o texto literal `"Erro"` no display. Comportamento detalhado:
   - Quando o usuário pressiona `÷` `0` `=` (ou `÷` `0` durante encadeamento), o display muda para `"Erro"` em vez de retornar `Infinity`, `NaN` ou crashar
   - **Estado de erro:** após exibir `"Erro"`, cliques em dígitos, vírgula, operadores e igual devem ser **ignorados** (no-op) — apenas o botão **Clear** restaura o estado normal
   - Pressionar **Clear** após `"Erro"` retorna ao estado inicial (display = `"0"`, operandos zerados, calculadora utilizável normalmente)
   [Source: docs/prd.md#Requirements — FR8 + docs/project-brief.md#decisoes-confirmadas-2]

7. **AC1.2.7:** O botão Clear (`data-action="clear"`) reseta:
   - O display ao valor `"0"`
   - O operando atual em digitação
   - O operando anterior (previousValue)
   - A operação pendente (pendingOperator)
   - O estado de erro (se ativo)
   - Qualquer flag interno de "aguardando segundo operando"

   Após Clear, a calculadora volta ao mesmo comportamento que tinha quando a página foi aberta inicialmente. [Source: docs/prd.md#Requirements — FR5]

8. **AC1.2.8:** **Suporte a teclado físico está fora do escopo do MVP.** A interação ocorre **exclusivamente via clique do mouse ou toque (touch)** nos botões. **Validação obrigatória:** pressionar teclas no teclado físico (números, operadores, Enter, Escape, Backspace, etc.) **não deve** alterar o estado da calculadora nem gerar erros no console. Implementação: simplesmente **não adicionar** event listeners de `keydown`/`keyup`/`keypress` no `document` ou `window`. [Source: docs/prd.md#Requirements — NFR7 + docs/project-brief.md#decisoes-confirmadas-3]

9. **AC1.2.9:** O código JavaScript da Story 1.2 deve estar **inline em `<script>` no `index.html`** OU em um arquivo `script.js` local na raiz do projeto (escolha do dev — mas deve seguir a estratégia adotada na Story 1.1, que foi **arquivo único `index.html` com tudo inline**). É **proibido**:
   - Carregar JS via CDN (`<script src="https://...">`)
   - Usar `import`/`require` de pacotes externos (sem `node_modules`)
   - Usar bundlers (Webpack, Vite, Parcel)
   - Usar TypeScript ou qualquer transpilação
   - Adicionar `package.json`, `package-lock.json` ou similares

   O código deve ser **JavaScript ES6+ vanilla** que rode nativamente em Chrome, Edge e Firefox (últimas 3 versões). [Source: docs/prd.md#Requirements — NFR1, NFR2 + Story 1.1 Completion Note #1]

### Diagrama de estados (state machine) — referência crítica

A regra mais sutil desta story é o **encadeamento ATIVO**. Aqui está o fluxo correto:

```
Estado inicial: { currentInput: "0", previousValue: null, pendingOperator: null,
                  waitingForSecondOperand: false, isErrored: false }

Cliques: 2 + 3 + 5 =

[clique 2]
  currentInput "0" → "2" (substitui zero inicial via AC1.2.1)
  estado: { currentInput: "2", ... }
  display: "2"

[clique +]
  pendingOperator é null → previousValue = parseFloat("2") = 2
  pendingOperator = "add"
  waitingForSecondOperand = true
  display: "2" (continua mostrando o operando anterior — AC1.2.3)

[clique 3]
  waitingForSecondOperand é true → currentInput = "3"; waitingForSecondOperand = false
  display: "3"

[clique +]  ← ENCADEAMENTO
  pendingOperator é "add" e waitingForSecondOperand é false
  → result = compute(2, 3, "add") = 5
  → previousValue = 5
  → currentInput = "5"
  → pendingOperator = "add" (novo operador)
  → waitingForSecondOperand = true
  display: "5" (resultado intermediário)

[clique 5]
  waitingForSecondOperand é true → currentInput = "5"; waitingForSecondOperand = false
  display: "5"

[clique =]
  pendingOperator é "add"
  → result = compute(5, 5, "add") = 10
  → currentInput = "10"
  → previousValue = null
  → pendingOperator = null
  → waitingForSecondOperand = false
  display: "10"  ← ✅ resultado correto
```

### Decisões fundamentais (não negociáveis — vindas do sponsor)

[Source: docs/project-brief.md#decisoes-confirmadas]

1. **Encadeamento ATIVO esquerda → direita SEM precedência.**
   - `2 + 3 × 4 =` retorna **`20`**, não `14`.
   - Multiplicação NÃO tem prioridade sobre soma. A calculadora avalia na ordem em que os botões são pressionados.
   - Esta é uma decisão consciente do sponsor para manter simplicidade do MVP. **NÃO IMPLEMENTAR** parser de expressão com precedência.

2. **Divisão por zero → string literal `"Erro"`** no display (não `Infinity`, não `NaN`, não exception).
   - Calculadora entra em estado de erro estável.
   - Apenas Clear (`C`) restaura uso normal.

3. **Sem teclado físico no MVP.** Apenas clique/touch. Não adicionar listeners de `keydown`/`keyup`/`keypress`.

### Convenção decimal pt-BR

[Source: docs/prd.md#Technical Assumptions]

- **Internamente** (state, parseFloat, toFixed): usar **ponto** `.` — formato JavaScript nativo
- **Visualmente** (display): converter para **vírgula** `,` na função `renderDisplay()`
- O botão de vírgula usa `data-key="."` (ponto interno), label visual `","` — **isto já está pronto na Story 1.1**

Exemplo:
```javascript
// Estado interno
state.currentInput = "1.5";

// Renderização
display.textContent = state.currentInput.replace('.', ',');
// → exibe "1,5" para o usuário
```

### Precisão de ponto flutuante

[Source: docs/prd.md#Technical Assumptions, brief RIS-1]

JavaScript tem precisão imperfeita: `0.1 + 0.2 === 0.30000000000000004`.
**Solução obrigatória** em `formatNumber()`:
```javascript
function formatNumber(n) {
    return parseFloat(n.toFixed(10)).toString();
}
// formatNumber(0.30000000000000004) === "0.3"
```

Aplicar `formatNumber()` antes de armazenar em `currentInput` após cada `compute()`.

### Edge cases a considerar (sem ser bloqueador, mas registrados)

1. **Pressionar `=` sem nenhuma operação pendente:** display fica inalterado (no-op explícito em Task 6).
2. **Pressionar operador duas vezes seguidas (ex: `5 + + 3`):** segundo operador substitui o pendente sem calcular. Tratado em Task 5 (waitingForSecondOperand check).
3. **Pressionar Clear repetidamente:** sempre retorna a estado inicial (idempotente). Tratado por `resetState()`.
4. **Resultado muito grande/pequeno (ex: `9999999 × 9999999`):** JavaScript pode converter para notação exponencial. Aceitar comportamento padrão de `Number.prototype.toString()` — sem tratamento especial no MVP. Se aparecer `1e+14` no display, está OK (não é regressão funcional).
5. **Negativo intermediário (ex: `5 − 10 = ` → `−5`):** `compute()` retorna `-5`, `formatNumber(-5)` retorna `"-5"`, display mostra `-5` (o sinal negativo é parte do número, sem separador de vírgula relevante). OK.
6. **Vírgula no estado inicial:** clicar `,` quando display é `"0"` → currentInput vira `"0."` (display mostra `"0,"`). Tratado em Task 4.
